Functional Description


Component Goals


The goal of this component is to cleanse and parse input address data in order to assign an address code (which is known as an address identification). Address code is an unique ID in the official czech address registry (RÚIAN).
Core logic of the component is delivered by set of "Address identifier" steps with predefined set of parsing rules however component also works with replacement dictionaries.
Besides standard values it also provides output address in the envelope-ready printable form.
The component also supports evidentiary numbers (in Czech "evidenční číslo").

Interface


Input:
Address defined in three attributes (representing three "address lines" which is usual address format in the Czech Republic). Not all attributes must be filled, though.

As a standard in the Czech Republic, first address line usually contains street or city district/part followed by number, second line usually contains city or city district/part and third line usually contains postal code.
But of course, address structure is complex and the copomnent can handle multiple situations when some of these information is missing (naturally when there is no street net, for example, or by mistake/error in ETL processes). Key fact is that input address lines must contain enough information to unambiguously find the address code in the register.

From this point of view this componet is capable of handling following scenarios:
  1. some input values are optional depending on the presence of the others, for example:
    • when street + number and city is specified then postal code is not required
    • when city district + number and postal code is unambiguous enough then city value is not required
  2. input values presence and mapping varies, for example:
    • all of the input address attributes are filled: street + number in the first address line, city in the second and postal code in the third
    • two of the input address attributes are filled whilst the third is empty: street + number in the first address line, city in the second
    • only the first input address attribute is filled whilst the rest is empty: city + street + number in the first address line


Output:
Set of standardized address attributes (including std_address_code): these attributes are filled only for identified addresses whilst for unedified address they are empty.
Best existing value of the address lines values. For identified addresses it includes standardized values in the envelope-ready printable form. For unidentified addresses it is a copy of input address lines.

Component Behaviour


First attempt to identify address takes places after basic pre-cleansing of input values:
city+street: remove of typical č.p., č.o. etc
zip: remove of non-digit content, remove of dummy values (e.g. 5 consecutive numbers)

If address was successfully identified or at least parsed in the attempt then final score and output values are set. Parsed but unidentified address is not further processed because most probably input information result into ambiguous address (e.g. numbers are not defined precisely) and there is no use in processing it in another round of parsing.

If address was not successfully parsed in the first attempt then the component applies possible replacements on input values in order to get rid of misspellings and typos. Application of the replacements is case and diacritic insensitive.
After the replacements the second attempt to identify address takes place.

If address was not successfully parsed in the second attempt then third attempt to identify address takes place only when input parameter "Use extended patterns" is true; this time the address is parsed against large amount of patterns with (sometimes) unusual address values combinations so there is a bigger chance to parse low quality input.
The fact is this parsing is resources and time consuming so this option should be used reasonably.
Basic pattern set is stored in DATA/ext/lkp/cz_address_identifier/address-cfg.xml or extended patterns in DATA/ext/lkp/cz_address_identifier/address-cfg_2.xml. How to properly configure patterns see the reference documentation of Address Identifier step.

Lookup Files and External Sources


Component uses lookups generated specific for address identifier steps, besides the component working with street, city, etc. replacements.
The most important external data source used for address identification is called RÚIAN (in czech: Registr územní identifikace, adres a nemovitostí) - Czech official address registry.

Summary of Testing Data


Testing data contains set of addresses, main aim is to cover maximum possible cases: address valid (no need to use replacement), address invalid but repairable, address invalid and unparsable, etc. Also different situations and combinations of input are covered in testing data.

Attribute Details

sco_address

Data quality score of address

Value Description
0 Valid value. No modification was done to the input value, it was correct on input.
< 10 000 Corrected value. Small modification was done to the input value (squeeze spaces, street number set, street number/land registry number swap, etc.) but resulting value is still valid.
< 1 000 000 Unsafe value. Address code was identified, but huge modification was done to the input value and resulting value can no longer be considered valid, for example change of street name or adding a street even it was not found on input.
< 5 000 000 Invalid value. Address code was not identified but at least input address was parsed.
< 10 000 000 Invalid value. Address code was not identified neither input address was parsed.
>= 10 000 000 Empty value of main address input values: both first and second address lines are empty.

ADDR_AI_AMBIG_CL Scored 500000. City and land registry number are not sufficient to uniquely identify address id.
ADDR_AI_AMBIG_CL_CD Scored 500000. City and land registry number with city district are not sufficient to uniquely identify address id.
ADDR_AI_AMBIG_CL_ZCD Scored 500000. City and land registry number with ZIP are not sufficient to uniquely identify address id.
ADDR_AI_AMBIG_CS_N Scored 500000. City and street with street number are not sufficient to uniquely identify address id.
ADDR_AI_AMBIG_SN_Q Scored 500000. Street and street number with city part are not sufficient to uniquely identify address id.
ADDR_AI_AMBIG_ZC_L Scored 500000. City and land registry number with ZIP are not sufficient to uniquely identify address id.

ADDR_AI_AMBIG_ZCS_N Scored 500000. Postal code, city and street with street number are not sufficient to uniquely identify address id.
ADDR_AI_ID_MISMATCH Scored 500000. One of the input fields does not correspond to the address code, usually caused by inconsistency between the street number and the land registry number.
ADDR_AI_UNDECIDED_LRN_SN Scored 500000. It is not possible to decide which combination of land registry number and street number is correct (probably there is another existing address point at the same street having the same number but of other type).
ADDR_AI_CITY_CHANGED Scored 10000. City value was changed during the cleansing process.
ADDR_AI_STREET_CHANGED Scored 10000. Street value was changed during the cleansing process (E.g. 'Třída 1. máje' (in the input) is actually named '1. máje' (the official name)

NOTE: this case may also reflect the situation in which the input data reflects updated information about the street network, but the dictionary in DQC contain obsolete information.
ADDR_AI_STREET_SET Scored 10000. The street was found in the lookup dictionary and set to the output.
ADDR_AI_STREET_SURPLUS Scored 10000. Locality does not contain a street network (determined from the input data and lookup dictionary) and therefore the input street information is redundant. NOTE: this case may reflect the situation in which the input data reflects updated information about the street network, but the dictionary in DQC contain obsolete information.

ADDR_AI_LRN_TYPE_CHANGED Scored 1000. Type of land registry number has changed. It either has been parsed as evidence number (číslo evidenční) and should have been a house number (číslo popisné) or vice versa.
ADDR_AI_SN_CHANGED Scored 1000. Street number was changed
ADDR_AI_ZIP_CHANGED Scored 1000. Postal code was changed
ADDR_AI_CITY_SET Scored 100. City was found in the lookup dictionary and set to the output.
ADDR_AI_LRN_SET Scored 100. Land registry number was found in the lookup dictionary and set to the output.
ADDR_AI_SN_SET Scored 100. Street number was found in the lookup dictionary and set to the output.
ADDR_AI_AI_SWAP_LRN_SN Scored 100. Land registry number and street number were switched during the identification as opposed to the found rule. The step found out that switching of the land registry number and street number would cause better address identification. If the identification finished successfully, the components were actually switched in the input. In case of unsuccessful identification, the components were switched in the input with a high probability. In both cases, the result components are switched in comparison with the input values.

STREET_NUM_PREFIX_REMOVED Scored 10. Special label (e.g. "č.p.","č.o") was removed from city value during the cleansing process.
STREET_REPL_PART Scored 10. Partial street replacement was applied on street value (e.g. "nám. Míru" => "náměstí Míru").
STREET_REPL_FULL Scored 10. Full street replacement was applied on street value (e.g. "Al. Noskove" => "Alžběty Noskové").
STREET_REPL_CITY_FULL Scored 10. Full city replacement was applied on street value (i.e. in fact, there is probably a city name in the street value).

lbl_address

Output value of address validity label. Indicates whether the address is valid, corrected or unknown.

Value Description
VALID No or minimal correction was done to the input value. DQ score is 0. The result address is valid.
CORRECTED_MINOR Small correction was done to the input value. DQ score is greater than 0 and less than 10 000. The result address is valid-corrected.
CORRECTED_MAJOR Huge correction was done to the input value. DQ score is greater than 10 000 and less than 10 000 000. The result address is unsafe.
UNKNOWN No correction was able to help identify the address. Address delivery point was not identified. The result address is unknown or invalid.
INSUFFICIENT_INPUT Unknown value. Address delivery point was not identified as there is probably no address value in the input or major input values are missing.
